# 小红书面经整理

使用上游 `xhs-cli` 的 `xhs` 命令搜索、读取和整理小红书面经。默认只执行搜索、读取、评论读取等只读操作。

## 执行红线

- 未确认 `xhs status --json` 返回已登录前，不要搜索小红书。
- 用户未明确指定面试类型或面试阶段时，直接按“面试类型全选、面试阶段全选”搜索，不要停下来要求用户手动选择。
- 不要默认从 Chrome 读取 Cookie；必须先询问用户当前哪个浏览器已登录小红书。
- 不要让用户在聊天中发送小红书密码、短信验证码、Cookie、Token 或完整账号凭据。
- 不要执行 `like`、`favorite`、`follow`、`comment`、`post`、`delete` 等写操作，除非用户明确要求。
- 报告正文不放引用编号；引用和异常统一放在末尾附录。

## 登录协议

每次执行面经搜索前，先检查登录状态：

```bash
xhs status --json
```

如果显示已登录，继续进入默认筛选与搜索。

如果未登录或状态不明确，先引导用户在本机浏览器打开并登录小红书网页版：

```text
https://www.xiaohongshu.com/
```

然后询问用户哪个浏览器已经登录小红书，并按用户选择读取 Cookie。常见命令如下：

```bash
xhs login --cookie-source chrome
xhs login --cookie-source firefox
xhs login --cookie-source edge
xhs login --cookie-source safari
xhs login --cookie-source brave
xhs login --cookie-source arc
```

如果用户不确定支持哪些浏览器，先查看帮助：

```bash
xhs login --help
```

如果用户尚未在浏览器登录，先让用户完成网页登录，再读取 Cookie。如果浏览器 Cookie 读取失败，使用二维码兜底：

```bash
xhs login --qrcode
```

每次登录尝试后必须再次运行：

```bash
xhs status --json
```

只有确认已登录后，才可以进入搜索。

## 默认筛选策略

筛选项默认不阻塞搜索。用户没有明确指定时，直接按以下默认值执行：

- 面试类型：全选
- 面试阶段：全选

用户主动指定筛选时，再收窄检索：

- 面试类型：全选 / 日常实习 / 暑期实习 / 秋招校招 / 社招正式
- 面试阶段：全选 / 笔试 / 群面 / 一面 / 二面 / 三面终面 / HR面

识别规则：

- 用户只说岗位或方向，例如“AI 产品经理面经”，按两个维度都全选。
- 用户提到“实习”“暑期实习”“秋招”“校招”“社招”等，映射到面试类型；未提到面试阶段时，阶段仍全选。
- 用户提到“一面”“二面”“三面”“终面”“HR 面”“笔试”“群面”等，映射到面试阶段；未提到面试类型时，类型仍全选。
- 用户说“默认”“全选”“不限”“都行”，按两个维度都全选。

执行前可以用一句话轻量告知默认筛选，但不要等待用户确认：

```text
我会按“面试类型全选、面试阶段全选”检索；如果你之后想收窄到实习/秋招/一面/二面等，我可以再补查。
```

只有当用户的表述确实无法判断搜索目标，例如没有岗位、方向、公司、行业或关键词时，才追问目标本身。

## 搜索与采集

根据用户目标和筛选项扩展 4-8 个搜索词。未指定筛选时使用全选语义，优先围绕岗位、方向、面经、面试、实习、校招、社招和轮次组合关键词；不要额外追问公司或城市筛选，除非用户主动给出。

示例：

```bash
xhs search "AI产品经理 面经" --json
xhs search "AI产品经理 面试题" --json
xhs search "AI产品经理 实习 面经" --json
xhs search "AI产品经理 一面 面经" --json
xhs search "AI产品经理 秋招 面经" --json
```

采集规则：

- 先保留较宽候选池，再按标题、摘要、正文、note ID、URL 去重。
- 优先读取高相关笔记详情：

```bash
xhs read NOTE_ID_OR_URL --json
```

- 只有当评论可能补充面试题、回答、轮次或岗位信息时，再读取评论：

```bash
xhs comments NOTE_ID_OR_URL --json
```

- 读取详情时优先使用 `xhs search` 返回的 URL 或标识，避免裸 note ID 缺少上下文 token 导致失败。
- 失败命令、读取失败、登录异常、样本不足都要记录到报告附录，不要把失败来源当作有效证据。

## 来源分类

读取后的来源按内容归类：

- 带参考回答的面经：原文包含明确问题和回答、答题思路、复盘答案或可还原的回答要点。
- 面试题清单：原文只列出面试问题，或以问题为主、没有明确回答。
- 流程/体验信息：原文主要是轮次、流程、时间线、岗位要求、面试体验，不含足够问题。
- 相关性不足：标题或正文与用户目标偏离，不进入正文总结，可在附录说明。

## 报告格式

最终报告必须输出结构化 Markdown，且必须包含以下四个二级标题：

```markdown
# 小红书面经整理报告

## 高频题库

## 带参考回答的面经

## 面试题清单

## 备战建议
```

四部分写法：

- 高频题库：基于本次提取到的所有面试问题分类整理，只包含问题，不写回答。可按产品思维、业务理解、AI/技术理解、数据分析、项目经历、岗位认知、HR/稳定性等类别分组。
- 带参考回答的面经：按来源整理原文中包含回答的问题，问题和回答成对呈现。尽量保留原文含义；如果是归纳回答，要说明是“整理要点”。
- 面试题清单：按来源整理只有问题、没有回答的内容。
- 备战建议：基于前三部分归纳行动建议，写成可执行准备清单。

正文不写 `[S1]`、`来源 1` 等引用标记。所有来源统一放入末尾附录。

## 可用 xhs 查询能力

优先使用这些只读命令：

- `xhs status --json`：检查登录状态。
- `xhs search "关键词" --json`：搜索笔记。
- `xhs read NOTE_ID_OR_URL --json`：读取单篇笔记详情。
- `xhs comments NOTE_ID_OR_URL --json`：读取笔记评论。
- `xhs sub-comments NOTE_ID COMMENT_ID --json`：读取评论回复。
- `xhs search-user "关键词" --json`：搜索用户。
- `xhs user USER_ID --json`：读取用户资料。
- `xhs user-posts USER_ID --json`：读取用户公开发布笔记。
- `xhs topics "关键词" --json`：搜索话题/标签。
- `xhs feed --json`：读取推荐流。
- `xhs hot --json`：读取热门/趋势内容。
- `xhs my-notes --json`：读取当前账号发布的笔记。
- `xhs notifications --json`、`xhs unread --json`：读取通知和未读数。

注意：`xhs favorites` 是上游收藏列表命令，但接口可能失败；上游没有稳定的收藏夹标签/分组查询能力。

## 登录与隐私

`xhs login --cookie-source <browser>` 读取的是用户本机浏览器登录态。前提是浏览器已经登录小红书网页版，且登录态未过期。如果 `xhs-cli` 调起本机浏览器窗口，用户可以在窗口内完成登录授权；这是本机流程，不应出现在报告或最终输出中。

如果出现验证码、风控、接口失败或登录过期，停止高频请求，并向用户说明当前限制和可选登录方式。
