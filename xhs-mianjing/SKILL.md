# 小红书面经整理

使用上游 `xhs-cli` 的 `xhs` 命令搜索、读取和整理小红书面经。

## 快速流程

1. 检查本机是否有 `xhs`：

```bash
xhs --help
```

2. 检查登录状态：

```bash
xhs status
```

3. 如果未登录，先引导用户在本机浏览器登录小红书网页版，再运行自动探测：

```bash
xhs login
```

如果自动探测失败，再按用户实际浏览器指定来源：

```bash
xhs login --cookie-source chrome
xhs login --cookie-source firefox
xhs login --cookie-source edge
xhs login --cookie-source safari
```

二维码兜底：

```bash
xhs login --qrcode
```

4. 根据用户的方向、岗位、城市、行业、公司和输出偏好，扩展 4-8 个关键词，先广泛搜索：

```bash
xhs search "AI产品经理 面经" --json
xhs search "AI产品经理 实习 面经" --json
xhs search "上海 AI产品经理 面经" --json
```

5. 保留较宽的候选池，按 note ID、URL、标题相似度和正文相似度去重。从高相关来源读取详情；只有当评论可能包含面试细节或澄清信息时再读评论：

```bash
xhs read NOTE_ID_OR_URL --json
xhs comments NOTE_ID_OR_URL --json
```

6. 将详情来源分类为“原文含问答对”“原文只有面试问题”“原文只有流程/体验”“相关性不足”，再输出带来源的 Markdown 报告。失败命令或上游不可用能力要在报告末尾说明。

## 可用 xhs 查询能力

优先使用这些只读命令：

- `xhs search "关键词" --json`：搜索笔记。
- `xhs read NOTE_ID_OR_URL --json`：读取单篇笔记详情。
- `xhs comments NOTE_ID_OR_URL --json`：读取笔记评论。
- `xhs sub-comments NOTE_ID COMMENT_ID --json`：读取评论回复。
- `xhs search-user "关键词" --json`：搜索用户。
- `xhs user USER_ID --json`：读取用户资料。
- `xhs user-posts USER_ID --json`：读取用户公开发布笔记。
- `xhs topics "关键词" --json`：搜索话题/标签。
- `xhs feed --json`：读取推荐流。
- `xhs hot --json`：读取热门/趋势内容。
- `xhs my-notes --json`：读取当前账号发布的笔记。
- `xhs notifications --json`、`xhs unread --json`：读取通知和未读数。

谨慎对待这些写操作：`like`、`favorite`、`unfavorite`、`follow`、`unfollow`、`comment`、`reply`、`post`、`delete`。

注意：`xhs favorites` 是上游收藏列表命令，但接口可能失败；上游没有稳定的收藏夹标签/分组查询能力。

## 搜索策略

根据用户输入组合关键词：

- `<方向> 面经`
- `<方向> 面试`
- `<方向> 实习 面经`
- `<方向> 秋招 面经`
- `<方向> 校招 面经`
- `<公司> <方向> 面经`
- `<城市> <方向> 面经`
- `<行业> <方向> 面经`

示例：

- `AI产品经理 面经`
- `AI产品经理 实习 面经`
- `AI产品运营 面经`
- `上海 AI产品经理 面经`
- `小红书 产品经理 面经`
- `券商 行研实习 面经`

探索型请求可覆盖岗位、行业和阶段，例如：

```bash
xhs search "金融分析师实习 面经" --json
xhs search "券商 行研实习 面经" --json
xhs search "基金 研究员 实习 面试" --json
```

定向请求可组合公司、城市和岗位，例如：

```bash
xhs search "小红书 AI产品经理 面经" --json
xhs search "上海 AI产品经理 实习 面经" --json
xhs search "小红书 产品经理 面试" --json
```

读取详情时优先使用 `xhs search` 返回的 URL 或标识，避免裸 note ID 缺少上下文 token 导致失败。

## 报告要求

输出结构化 Markdown，不要只堆链接。最终报告必须包含以下 4 个一级内容：

- 高频面试问题：基于本次提取到的所有面试问题做分类整理，只保留问题，不写回答。可按产品、业务、项目、技术理解、数据分析、HR 等类别分组。
- 问答对：按来源整理原文中包含回答的问题。问题和回答成对呈现，尽最大可能保持原文完整性，并标注来源编号。
- 仅问题：按来源整理原文只给出问题、没有回答的内容，并标注来源编号。
- 准备建议：基于高频面试问题、问答对和仅问题归纳行动建议，写成可执行的准备清单。

其他信息按需放在最终报告末尾，例如搜索说明、面试轮次、公司/岗位线索、能力要求、来源列表和采集异常。每个重要结论尽量标注来源编号，明确区分“原文提到”和“归纳判断”。

## 登录与隐私

不要让用户在聊天中发送小红书密码、短信验证码、Cookie、Token 或完整账号凭据。

`xhs login` 读取的是用户本机浏览器登录态。前提是浏览器已经登录小红书网页版，且登录态未过期。如果 `xhs-cli` 调起本机浏览器窗口，用户可以在窗口内完成登录授权；这是本机流程，不应出现在报告或最终输出中。

如果出现验证码、风控、接口失败或登录过期，停止高频请求，并向用户说明当前限制和可选登录方式。
